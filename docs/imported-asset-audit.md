# Imported Asset Audit

## Scope

This audit focuses on [public/imported](public/imported) as of the current split-history state on `main`.

## Verified Findings

- `public/imported` contains `3303` tracked files.
- Almost the entire tree lives under `public/imported/kilogrammes/wp-content/uploads/YYYY/MM/`, which matches a mirrored WordPress media-library structure.
- The repository object store is already about `476 MiB`, and imported binaries are the primary reason.
- A small number of assets are directly referenced by active catalog/content data and snapshot scripts.

## Active References Seen In The Repo

Direct references were verified in:

- [src/data/catalog/categories.json](src/data/catalog/categories.json)
- [src/data/catalog/variations.json](src/data/catalog/variations.json)
- [src/data/content/pages.json](src/data/content/pages.json)
- [scripts/snapshot-kilogrammes-catalog.mjs](scripts/snapshot-kilogrammes-catalog.mjs)
- [scripts/lib/kilogrammes-media.mjs](scripts/lib/kilogrammes-media.mjs)

Representative active assets include:

- category and terp icons in `2022/11`, `2022/12`, and `2024/02`
- current product visuals in `2026/01`, `2026/02`, and `2026/03`
- editorial/franchise images in `2024/02`, `2024/03`, and `2025/06`
- the PDF `public/imported/kilogrammes/shop_kilogrammes.pdf`

## What Should Stay Versioned For Now

Keep these subsets in Git history for now because they are actively consumed at build time or runtime:

- product images currently referenced by [src/data/catalog/variations.json](src/data/catalog/variations.json)
- category imagery referenced by [src/data/catalog/categories.json](src/data/catalog/categories.json)
- content-page assets embedded inside [src/data/content/pages.json](src/data/content/pages.json)
- media overrides referenced in [scripts/snapshot-kilogrammes-catalog.mjs](scripts/snapshot-kilogrammes-catalog.mjs)
- the public PDF `public/imported/kilogrammes/shop_kilogrammes.pdf`

## Best Candidates To Stop Versioning Later

These look like the safest first reduction targets once a replacement workflow exists:

1. legacy `2019` to `2021` WordPress uploads that are not referenced by current snapshots or content
2. oversized blog/editorial stock images, especially `*-unsplash.webp`
3. WordPress-generated thumbnail variants such as `-150x150`, `-300x300`, `-600x600`, `-768x*`, `-1024x*`
4. duplicate historical image sizes that can be regenerated by a build-time pipeline

## Recommended Smaller Committed Set

The most pragmatic target is not "remove all imported media". It is:

1. keep only assets that are referenced by current catalog/content JSON or hard-coded snapshot overrides
2. keep a very small hand-maintained subset of branded files that are not reproducible elsewhere
3. stop committing mirrored WordPress thumbnails and stale historical uploads once regeneration or re-sync is in place

## Safe Migration Order

1. install Git LFS locally
2. keep future imported binaries under LFS via [.gitattributes](.gitattributes)
3. add a repeatable script or report that enumerates currently referenced imported assets
4. prune unreferenced legacy folders in a dedicated branch and validate the build
5. only consider a full historical rewrite if repository growth remains a problem after steps `1` to `4`

## Risks If You Reduce Too Early

- broken product images where snapshots still point at removed files
- broken content sections where raw HTML inside [src/data/content/pages.json](src/data/content/pages.json) still references WordPress-sized variants
- broken snapshot scripts if hard-coded media overrides are removed without replacement

## Practical Conclusion

For this repo, the right immediate move is to stop future growth first, not to mass-delete assets immediately. Once future binaries are under LFS, the next cleanup pass should target old mirrored uploads and WordPress thumbnail variants rather than current product or content assets.
